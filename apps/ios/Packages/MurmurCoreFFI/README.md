# MurmurCoreFFI

Local Swift package wrapping the `crates/ffi` UniFFI bridge (Plan 07,
`docs/plans/2026-07-04-rust-core-07-ffi-bridge.md`, Task 9).

## Layout

| Path | Committed? | What |
|---|---|---|
| `Package.swift` | yes | binaryTarget (`ffiFFI`) + Swift target (`MurmurCoreFFI`) |
| `Sources/MurmurCoreFFI/ffi.swift` | yes | generated by `uniffi-bindgen`; regenerate via `build-ffi.sh` whenever `crates/ffi`'s exported surface changes |
| `Frameworks/ffiFFI.xcframework` | **no** (gitignored) | device (`ios-arm64`) + simulator (`ios-arm64-simulator`) static-lib slices; large binary, machine-specific, fully reproducible |

## Regenerating

From the repo root:

```
apps/ios/build-ffi.sh
```

This builds `crates/ffi` for `aarch64-apple-ios-sim` and `aarch64-apple-ios`,
regenerates the Swift bindings, and rebuilds `Frameworks/ffiFFI.xcframework`.
Run it once after cloning (the xcframework is gitignored) and again any time
`crates/ffi`'s public surface changes.

## Why the build isn't plain `nix develop -c cargo build --target ...`

`flake.nix`'s dev shell (Plan 07 Task 1) provides a rust-overlay toolchain
with the `aarch64-apple-ios{,-sim}` targets installed, so `rustc`/`cargo` can
compile Rust for iOS. But the nix-wrapped `cc`/linker hardcodes macOS SDK
paths (`-mmacos-version-min`, and library search that resolves against the
MacOSX SDK even when `--target` is an iOS triple), which fails at the final
link step:

```
ld: building for iOS Simulator, but linking in .tbd built for macOS/Mac Catalyst,
file '.../MacOSX.platform/.../MacOSX.sdk/usr/lib/libiconv.tbd' for architecture arm64
```

The fix (used by `build-ffi.sh`): keep the nix devShell's toolchain for
`cargo`/`rustc`/`clippy` (host builds, `cargo test --workspace`, CI —
unchanged), but for the iOS cross-compiles specifically, point `CC`/`AR`/the
linker at the **system Xcode** toolchain (`/usr/bin/clang`, `/usr/bin/ar`)
and `SDKROOT` at the real iphoneos/iphonesimulator SDK via `xcrun`, and unset
the nix-injected `NIX_*FLAGS`. This is the "system Xcode fallback" — a
pure-nix path (rust-overlay toolchain + nix's own clang/linker end to end)
was tried first and hits the link failure above.

## Reference

`crates/ffi/uniffi-bindgen.rs` is the dev-only `uniffi-bindgen` CLI entry
point (`cargo run -p ffi --features uniffi-bindgen-cli --bin uniffi-bindgen
-- generate ...`), gated behind the `uniffi-bindgen-cli` feature so
`uniffi`'s `cli` feature never lands in the shipped cdylib/staticlib.
